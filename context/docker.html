Docker Desktop — Local Development Context

This file tells agents how Docker Desktop is used in this project for local development, running services, and E2E testing. Agents (backend, frontend, qa, devops) read this alongside tech-stack.html.

Prerequisite: Docker Desktop must be running before executing any docker or docker compose commands.

E2E Browser Testing via Docker

The qa agent runs Playwright (or Cypress) tests against the locally running Docker Compose stack. Services must be healthy before tests run.

Running E2E tests locally

# 1. Ensure Docker Desktop is running and services are up
docker compose up -d

# 2. Wait for services to be healthy
docker compose ps   # all services should show "healthy" or "running"

# 3. Run E2E tests (adjust to your test framework)
npx playwright test           # Playwright
npx cypress run               # Cypress

Running E2E tests inside Docker (CI parity)

# Playwright in Docker (no display needed — runs headless)
docker run --rm --network host -v $(pwd):/app -w /app \
  mcr.microsoft.com/playwright:v1.x.x-jammy \
  npx playwright test
The qa agent will use the run commands from tech-stack.html (E2E / browser row). For browser-headed runs on Windows, Playwright uses the system Chrome/Chromium installed by Docker Desktop or the local Node install.

Database Migrations

# Run migrations against the Docker Compose db service
docker compose exec api <migration-command>

# Examples by framework:
# Flyway:     docker compose exec api ./mvnw flyway:migrate
# Alembic:    docker compose exec api alembic upgrade head
# Prisma:     docker compose exec api npx prisma migrate deploy
# Raw SQL:    docker compose exec db psql -U postgres -d mydb -f /migrations/V001__init.sql

The devops agent will generate matching HEALTHCHECK directives in Dockerfiles and readiness/liveness probes in deployment manifests based on these conventions.

Image Build Conventions

  • All Dockerfiles are multi-stage: builder → runtime.
  • Runtime stage uses a minimal base image (e.g. node:22-alpine, python:3.12-slim).
  • Containers run as a non-root user — never as root.
  • No secrets baked into images. Use environment variables or secret mounts at runtime.
  • Images are tagged with git SHA (:sha-abc1234) — never :latest.
