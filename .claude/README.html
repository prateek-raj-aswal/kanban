.claude — Lean SDLC Agent System

A 9-agent / 13-command pipeline organized around a 5-stage lifecycle, with declarative per-agent harnesses, domain + per-story memory for context optimization, and automatic event logging via a PostToolUse hook.

2. The 5-Stage Lifecycle

#StagePrimary agent(s)Output (handoff to next stage)
1CLARIFYclarify (INTERROGATE → PLAN)requirements.html + plan.json + kanban.json
2DESIGNarchitect (DESIGN → EXTRACT), gated by auditor REVIEWarchitecture/system-design.yaml (PASS) + stories/{id}/contracts.json
3BUILDbackend + frontend + qa + auditor REVIEW + doc-writer (per story)repo code + stories/{id}/* artifacts
4VERIFYauditor SECURITY (project-wide)security/findings.json (BLOCKED or CLEAR)
5SHIPdevops + doc-writer (final) + qa REGRESSION (gate) + auditor DELIVERYDockerfile, manifests, CI/CD, docs/, regression HTML reports, delivery.html

Gates between stages are implicit when you run stage commands individually. Inside /full-pipeline there are 3 explicit human gates: after CLARIFY (plan approval), after VERIFY (only if BLOCKED), end of SHIP (GO/NO-GO).

3. Agent Reference (9)

3.1 tracker (cross-cutting)

ModesINIT, MOVE, STATUS, NEXT
Writeskanban.json
ToolsRead, Write, Edit
  • INIT — create fresh board from a story list.
  • MOVE — advance one story (todo → in_progress → done; forward only).
  • STATUS — read and return current board.
  • NEXT — return stories whose dependencies are all done. Detects DEADLOCK.

3.2 clarify (Stage 1)

ModesINTERROGATE, PLAN
Writesrequirements.html · plan.json
ToolsRead, Write, Edit, Grep, Glob
Context readscontext/project-brief.html
  • INTERROGATE — adversarial alignment. Emits: ALIGNED | NEEDS_CLARIFICATION | REJECT.
  • PLAN — requires requirements.html with verdict ALIGNED. Produces PRD + vertical-slice phases + story backlog. Calls tracker INIT.

3.3 architect (Stage 2)

ModesDESIGN, EXTRACT
Writesarchitecture/system-design.yaml · stories/{id}/contracts.json
ToolsRead, Write, Edit, Grep, Glob
Context readscontext/tech-stack.html · context/constraints.html · context/project-brief.html
  • DESIGN — full architecture (components, APIs, DB, events). Every decision cites a requirement.
  • EXTRACT — slices design into per-story contracts. Self-resolves CONTRACT_GAPs.

3.4 backend (Stage 3)

ToolsRead, Write, Edit, Grep, Glob, Bash
Context readscontext/tech-stack.html

Implements backend code, versioned SQL migrations (UP+DOWN), and unit tests for one story. Runs tests via Bash before reporting done.

3.5 frontend (Stage 3)

ToolsRead, Write, Edit, Grep, Glob, Bash, WebFetch
Context readscontext/tech-stack.html · context/docker.html

Builds UI components and routing for one story. Integrates with APIs exactly as defined in contracts. After building, starts the dev server and uses WebFetch to confirm each page returns HTTP 200.

3.6 qa (Stage 3 + 4)

ModesPLAN, RETEST, REGISTER, REGRESSION
ToolsRead, Write, Edit, Grep, Glob, Bash, WebFetch
Context readscontext/tech-stack.html · context/docker.html
  • PLAN — writes executable test code (red phase). Unit, integration, and E2E browser tests. E2E targets local Docker Compose stack.
  • RETEST — executes tests via Bash with HTML reporters. Generates test-reports/{story_id}-backend.html and -frontend.html. Appends new bugs to bugs/registry.html.
  • REGISTER — after story passes + is done, promotes test files + run commands into tests/regression/suite.json. Refuses if bugs remain.
  • REGRESSION — at /ship time, runs every registered story's tests, generates aggregate HTML reports, emits PASS/FAIL verdict. A single failure blocks delivery.

3.7 auditor (Stages 3, 4, 5)

ModesREVIEW, SECURITY, DELIVERY
Writesstories/{id}/review.html · architecture/design_review.html · security/findings.json · delivery.html
ToolsRead, Write, Edit, Grep, Glob (no Bash)
  • REVIEW — PASS | FAIL (soft signal). Five sections: Summary, Critical, Warnings, Suggestions, Verdict.
  • SECURITY — STRIDE + OWASP Top 10. Verdict: BLOCKED | CLEAR.
  • DELIVERY — final GO/NO-GO: APPROVE | APPROVE WITH CONDITIONS | REJECT.

3.8 devops (Stage 5)

ToolsRead, Write, Edit, Grep, Glob, Bash
Context readscontext/tech-stack.html · context/docker.html

Generates Dockerfiles (multi-stage, non-root), deployment manifests (K8s / ECS / Cloud Run / Docker Compose), CI/CD pipeline files, and health checks. Aligns Docker Compose services with the conventions established in context/docker.html.

3.9 doc-writer (Stages 3 + 5)

ToolsRead, Write, Edit, Grep, Glob
Writesdocs/README.html · docs/api/{resource}.html · docs/adr/ADR-{N}.html · docs/runbooks/{feature}.html · docs/stories/index.html

Reads existing docs first; makes surgical edits only. API docs include full field-level schemas (required/optional, format constraints, validation rules) and all error codes with trigger conditions. Maintains docs/stories/index.html — a persistent, append-only registry of every story. ADRs only for user-selected decisions.

5. Harness Format

Every agent has a sibling harness at .claude/harnesses/{name}.harness.yaml. The agent reads it as the first thing it does. Scope violations are refusals, not silent successes.

agent: {name}
purpose: |
  One-paragraph contract...

modes:
  MODE_NAME:
    inputs:   [...]
    outputs:  [...]
    prerequisites:  [...]
    error_modes:    [ERROR_NAME: explanation]

memory:
  reads:  [...]    # the ONLY paths this agent will read
  writes: [...]    # the ONLY paths this agent may write

tools_allowed: [Read, Write, Edit, Grep, Glob, Bash, WebFetch]
can_invoke: [other-agent-name]
ConcernHow the harness handles it
Context bloatmemory.reads is a closed list — no "everything in /memory" dumps
Tool drifttools_allowed mirrors frontmatter tools: — Claude Code enforces it hard
Stage handoffprerequisites make every agent self-policing
Hidden couplingcan_invoke documents the full dependency graph
Debugging vocabularyerror_modes give every refusal a greppable name

7. Event Logging Hook

Hook scripts

Hook eventScriptWhat it captures
PostToolUse (Write|Edit|Bash)log-event.ps1Every file mutation or shell command — agent + story attribution
Stoplog-usage.ps1Token usage at end of every agent turn — context vs output breakdown

settings.json config

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit|NotebookEdit|Bash",
        "hooks": [{ "type": "command", "command": "powershell.exe ... log-event.ps1" }]
      }
    ],
    "Stop": [
      {
        "hooks": [{ "type": "command", "command": "powershell.exe ... log-usage.ps1" }]
      }
    ]
  }
}

Log entry shapes

Tool event (from log-event.ps1, event field absent = tool event):

{"ts":"2026-05-28T10:23:14Z","agent":"qa","story_id":"US-003","tool":"Write","path":".claude/memory/stories/US-003/test_plan.yaml","ok":true}
{"ts":"2026-05-28T10:24:01Z","agent":"backend","story_id":"US-003","tool":"Bash","command":"npx jest --testPathPattern=registration","ok":true}

Job completion (from log-usage.ps1, "event":"job_complete"):

{
  "ts":        "2026-05-28T10:25:00Z",
  "event":     "job_complete",
  "agent":     "backend",
  "story_id":  "US-003",
  "tokens": {
    "context":     42150,   // total input  = input_new + cache_new + cache_read
    "output":       2840,   // tokens the model generated
    "total":       44990,
    "input_new":   18200,   // uncached input (full price)
    "cache_new":    5400,   // written to cache this turn (1.25× price, amortised)
    "cache_read":  18550,   // served from prompt cache (~0.1× price)
    "context_pct":   93.7  // context / total — how saturated the window is
  }
}

Token field glossary

FieldMeaningCost note
contextTotal tokens sent IN (context window)Sum of the three input rows
outputTokens the model generatedTypically 3–15× more expensive per token than input
input_newFresh, uncached input tokensFull input price
cache_newTokens written into a new cache entry~1.25× input price (write cost)
cache_readTokens served from existing cache~0.1× input price — very cheap
context_pctcontext / total × 100~90–97 % is normal for large agents

Querying the log

# ── Token usage summary per agent across all jobs ──────────────────────────────
Get-Content .claude/memory/events.jsonl |
  ForEach-Object { $_ | ConvertFrom-Json } |
  Where-Object   { $_.event -eq 'job_complete' } |
  Group-Object   { $_.agent } |
  ForEach-Object {
    $jobs = $_.Group
    [PSCustomObject]@{
      Agent       = $_.Name
      Jobs        = $jobs.Count
      TotalTokens = ($jobs | Measure-Object { $_.tokens.total }   -Sum).Sum
      AvgContext  = [Math]::Round(($jobs | Measure-Object { $_.tokens.context_pct } -Average).Average, 1)
      CacheRead   = ($jobs | Measure-Object { $_.tokens.cache_read } -Sum).Sum
    }
  } | Format-Table -AutoSize

# ── All job completions for one story ──────────────────────────────────────────
Get-Content .claude/memory/events.jsonl |
  ForEach-Object { $_ | ConvertFrom-Json } |
  Where-Object   { $_.event -eq 'job_complete' -and $_.story_id -eq 'US-003' } |
  Select-Object  ts, agent, @{n='total';e={$_.tokens.total}}, @{n='ctx%';e={$_.tokens.context_pct}}

# ── Tool mutations (file writes/edits) for story US-003 ────────────────────────
Get-Content .claude/memory/events.jsonl |
  ForEach-Object { $_ | ConvertFrom-Json } |
  Where-Object   { -not $_.event -and $_.story_id -eq 'US-003' } |
  Select-Object  ts, agent, tool, path, command

8. Common Workflows

8.1 One-shot (autonomous, 3 gates)

/full-pipeline
  └── Gate 1 — review & approve plan
  └── (autonomous design + build with per-story TDD)
  └── Gate 2 — if security BLOCKED, fix and re-verify
  └── Gate 3 — APPROVE / APPROVE WITH CONDITIONS / REJECT

8.2 Deliberate stage-by-stage

/clarify          → review requirements + plan
/design           → review architecture
/build            → review delivered stories
/verify           → review security findings
/ship             → GO/NO-GO

8.3 Surgical iteration on one story

/build US-003          # one story only
/run-tests US-003      # re-run tests after a fix
/review-code US-003    # re-review after edits

8.4 Resuming after a break

/board                 # see todo / in_progress / done
/next                  # see what's unblocked right now
/execute-story US-007  # pick up a ready story

8.5 Recovering from a TDD escalation (5-iter cap)

  1. Inspect .claude/memory/stories/{id}/bugs.json for the latest failing tests.
  2. Inspect .claude/memory/stories/{id}/review.html for the auditor's analysis.
  3. Decide: fix the design (/design {id}), split the story, or manually override (tracker MOVE to done).
